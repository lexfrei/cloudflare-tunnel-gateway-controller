# GatewayClass SHOULD / MAY assessment (GC-*)

Architecture: GatewayClass is bound by `spec.controllerName`; the status-only `GatewayClassReconciler` (`internal/controller/gatewayclass_controller.go`) writes the `Accepted` and `SupportedVersion` conditions. The Cloudflare credentials / tunnelID come from a `GatewayClassConfig` resolved via `parametersRef` by `internal/config/resolver.go`. Verdicts below were re-derived from the live code, not from `rows-GC.md`.

Documented in limitations.md ("The gateway-exists finalizer is managed"). |

## MAY clauses

| ID | Verdict | Evidence | Notes |
| --- | --- | --- | --- |
| GC-13 | OMITTED-INTENTIONAL | gatewayclass_controller.go:156-162, 212-217 | Best-effort support for an unrecognised CRD version. The controller does NOT pursue this option: on any major.minor skew it deterministically reports SupportedVersion=False/UnsupportedVersion rather than attempting partial support. It does, however, leave Accepted=True (does not hard-reject), so it neither claims best-effort support (GC-13) nor refuses outright (GC-14) — it signals skew and keeps running. Choosing not to provide best-effort support is a valid MAY stance. |
| GC-14 | OMITTED-INTENTIONAL | gatewayclass_controller.go:104-114, 156-162 | The alternative "refuse to support unrecognised CRD versions" is also not taken in the strong sense: Accepted stays True even on an unsupported bundle, so Gateways are still provisioned. The controller's actual posture is "warn via SupportedVersion=False, keep serving" — neither MAY option is selected as written; the SHOULD signal (GC-15) carries the operator warning instead. Valid: both GC-13 and GC-14 are MAY, so declining both is conformant. |
